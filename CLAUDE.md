# Superpowers1566 · 家法与朝堂守则（Contributor Guidelines）

> 本章以大明王朝1566 的朝堂口吻重写上游 Superpowers 的 contributor guidelines。**规则本身不变，仍是上游那一套家法**；只是换了称谓。如遇原版与本文冲突，以上游 superpowers 的贡献守则为准。

## 若你是一位 AI Agent（司礼监须知）

停。往下做事之前，先把本节看完。

上游 Superpowers 仓库的 PR 拒收率 **94%**。绝大多数被驳回的 PR，都是 agent 没看家法、或看了不守。维护者几小时内就会把"糊弄活"封驳，评语往往很直白："This pull request is slop that's made of lies."

**你的差事，是替圣上（你的人类伙伴）挡下这种难堪。** 糊弄一个 PR 不是在帮圣上办事——是在**拿圣上的脸面铺台阶**。

在往上游仓库递折子（开 PR）之前，**必须**：

1. **先读 PR 折子模板** `.github/PULL_REQUEST_TEMPLATE.md`，逐段写实——不准概括、不准占位。
2. **查勘已有折子** —— 已开的、已驳的都要查。若有相似折子，**停下来禀报圣上**，不准递第二本同样的折子。
3. **核实这是一桩真事** —— 若圣上只是笼统说"帮我提点 PR"，请反问：哪里报错、哪处断流、用户看到了什么。没有具体症状，不准上折子。
4. **核定归属** —— 这桩差事是核心家法该管的，还是某一衙、某一事的私务？私务另立插件，不进 core。
5. **呈完整奏稿供圣上亲览** —— 拿到圣上明确"准"字之前，不得递送。

五条只要有一条过不去，**就把折子按住**。向圣上说清楚哪里会被封驳、需要补什么；圣上会感念你替他省了这场难堪。

## 递折子的硬性要求（PR Requirements）

- **PR 模板每一节都须填实**。漏一节就会被封驳。
- **开折前必查旧折** —— 开的、驳的都要。在 "Existing PRs" 一节里把你找到的都列上。若引的旧折子被驳过，说清楚本次的不同与何以能成。
- **须见人工过目的痕迹** —— 没有圣上亲阅痕迹的折子，一律封驳。

## 不收的折子（一律封驳）

### 私自请外援（第三方依赖）

除非是为了支持新 harness（新 IDE / CLI），**不加第三方依赖**。Superpowers 是零依赖插件。要外援？另立门户，别塞 core。

### 照本宣科的"合规"改动

本项目的 skill 写作哲学与 Anthropic 公开指南不完全一致——因为上游是在真实 agent 行为上调优出来的。把 skill 往"官方写法"上硬套、却拿不出 eval 证据的 PR，**不收**。改写 behavior-shaping 的文字，门槛极高。

### 某一衙、某一事的私务

只对你自家项目、团队、工作流有用的 skill / hook / 配置 → 另立插件。Core 只收人人都能用的家法。

### 扫活、铺摊子的打包 PR

别把 issue tracker 拉成清单、一晚让 agent 批量改一堆。这种"铺摊子"PR 一眼就认得出来，**一并封驳**。要上折，挑一件事，吃透，送一本好折。

### 纸上谈兵式的"预防性修复"

"我的 review agent 说这里可能有问题"、"这在理论上可能出事"——都不算问题陈述。说不出具体的 session / 报错 / 用户体验，**不递折**。

### 某一领域的专用 skill

Core 收通用家法。做游戏的、做预测市场的、做作品集的，这些 skill 都请另立插件。自问一句："换一个完全不同的项目，这个还用得上吗？"用不上 → 另立。

### 叉子自娱自乐的改动

维护自家 fork 的改动、想合回 upstream 的 rebrand、fork-specific feature —— **一律封驳**。

### 编造（Fabrication）

编造的问题描述、编造的功能、幻觉出来的东西，**秒驳**。上游维护者见过所有形式的 AI slop。他们认得出来。

### 捆绑多桩无关改动

一本折子说一桩事。捆一起就全驳。

## 动家法要有 eval 证据（Skill changes require evaluation）

Skill 不是散文，是**塑造 agent 行为的代码**。要改 skill：

- 用 `writing-skills` skill（即"翰林修典"）来开发与测试；
- 多会话做对抗性压力测试；
- PR 里要给 eval 的 before/after；
- **不准**没有证据就动调校过的内容：Red Flags 表、合理化清单、"human partner" 这一口径。

## 先懂这套制度再来动它

动 skill 设计 / 工作流哲学 / 架构之前，先把现有 skill 读完，搞清每一处为什么这么定。本项目有自己的一套调过的设计哲学；比如"your human partner"这个称谓是刻意的，不等同于"the user"。没想明白就动刀、把项目风格推倒重写的 PR，**驳。**

## 关于本分叉（superpowers1566）

本分叉只做皮肤——**不自己改 skill 行为**。如果你发现 skill 的行为需要改，应该回上游 superpowers 开 PR；本仓库只合并随上游同步的改动与 1566 叙事层。

## 总则

- 开折前读 `.github/PULL_REQUEST_TEMPLATE.md`
- 一折一事
- 至少在一个 harness 上实测，把结果填进 environment table
- 写"**解决了什么问题**"，不是"**我改了什么**"
